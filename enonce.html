Contexte & mission

Vous êtes en binôme et venez d'être affectés à un nouveau projet pour le compte d'un client fictif : l'Automobile Club de Castellet, organisateur d'un Grand Prix de Formule 1. Il leur faut un système de réservation de places en tribune pour le weekend de course. Le CTO a été clair :

« On a déjà donné dans le code qui marche mais qu'on ne peut pas maintenir. Cette fois, je veux un livrable qui passe le Quality Gate Sonar Way par défaut avant tout déploiement. Sinon, on ne le déploie pas. »

Votre mission : démarrer un projet from scratch, implémenter le cahier des charges fonctionnel ci-dessous, et faire en sorte que la qualité soit pilotée par Sonar dès le premier commit, pas en toute fin de parcours.

  • Produire un livrable professionnel : code + tests + rapport synthétique.

Cahier des charges — Réservation GP F1

Vous développez le back-end (API REST ou CLI) d'un système de réservation de places en tribune pour un Grand Prix de Formule 1. Le weekend de course s'étale sur 3 jours (essais libres vendredi, qualifications samedi, course dimanche) avec plusieurs tribunes autour du circuit, chacune ayant sa propre catégorie tarifaire.

Le système doit gérer le catalogue (tribunes + sessions), les spectateurs, et le cycle de réservation / annulation avec calcul tarifaire et règles d'éligibilité. Pas de front-end requis — votre interface est l'API (ou la ligne de commande).

Entités métier (minimum)

Grandstand (Tribune)
  • id, nom (ex : "Tribune Sainte-Beaume"), localisation (ex : "Virage 3")
  • catégorie : BRONZE / SILVER / GOLD / PLATINUM
  • capacité totale (nombre de sièges)
  • prix de base par siège (€) — propre à la tribune
  • couverte : oui/non
Session
  • id, jour (FRIDAY / SATURDAY / SUNDAY), date
  • type : PRACTICE / QUALIFYING / SPRINT / RACE
  • multiplicateur de prix appliqué à la tribune
Spectator (Spectateur)
  • id, nom, email, date de naissance
  • programme fidélité : NONE / SILVER / GOLD
  • date d'inscription
Reservation
  • id, spectatorId, grandstandId, sessionIds[]
  • nombre de places, prix total calculé (€)
  • statut : CONFIRMED / CANCELLED
  • date de réservation, date d'annulation (nullable), montant remboursé (€)

Fonctionnalités obligatoires (7)

# Action métier Endpoint REST (exemple) / commande CLI
1Créer une tribunePOST /grandstands
2Lister/filtrer les tribunes (par catégorie)GET /grandstands?category=
3Créer une session (jour + type + multiplicateur)POST /sessions
4Inscrire un spectateurPOST /spectators
5Simuler le prix d'une réservation (sans la créer)POST /reservations/quote
6Créer une réservation (applique toutes les règles de prix & éligibilité)POST /reservations
7Annuler une réservation (calcule le remboursement)POST /reservations/{id}/cancel

Règles métier (toutes obligatoires)

  1. Calcul du prix de base : prix_tribune × multiplicateur_session × nb_places, sommé sur toutes les sessions de la réservation.
  2. Multiplicateurs par défaut : PRACTICE = ×0.5, QUALIFYING = ×1.0, SPRINT = ×1.2, RACE = ×1.8.
  3. Pass weekend : si la réservation couvre les 3 jours (FRIDAY + SATURDAY + SUNDAY) sur la même tribune → remise de 20%.
  4. Programme fidélité : remise additionnelle (cumulable après le pass weekend) — SILVER = 5%, GOLD = 10%.
  5. Tarif jeune : spectateur de moins de 16 ans → −50% sur le total.
  6. Limite par réservation : maximum 6 places par réservation.
  7. Disponibilité : on ne peut réserver que si places_libres ≥ nb_places_demandées sur chaque session de la tribune visée.
  8. Politique d'annulation : remboursement calculé à partir de la date de la première session réservée —
    • > 7 jours avant : 100% remboursé
    • < 7 jours : 0%
  9. Unicité & validation : email unique chez les spectateurs, format email valide, date de naissance < date du jour, multiplicateur de session > 0.
Note : ces règles sont volontairement croisées (type session × âge × fidélité × pass weekend) pour produire de la complexité métier — donc de la matière à structurer proprement, à tester unitairement et à factoriser pour éviter les god functions.

Contraintes techniques

Stack & outils

Méthode imposée — "Sonar dès le premier commit"

Principe : Clean as You Code, pas Clean at the End

L'erreur classique consiste à tout coder d'abord, puis à brancher Sonar à la fin et à éteindre des incendies dans l'urgence. Ici on fait l'inverse : vous configurez Sonar avant d'écrire la première ligne métier, et vous rescannez régulièrement pour piloter la qualité en continu.

Concrètement, votre boucle de travail tient en 5 étapes répétées :

  1. Je code une feature (ou un morceau).
  2. J'écris ses tests dans le même commit.
  3. Je commit avec un message clair.
  4. Je rescanne avec Sonar régulièrement.
  5. Si une issue apparaît, je la corrige immédiatement, avant la feature suivante.

Pourquoi cette discipline ?

Le Quality Gate Sonar Way ne juge que le code nouveau ou modifié. Si vous laissez s'accumuler les issues, vous aurez à les corriger toutes d'un coup dans l'urgence. En corrigeant au fur et à mesure, vous n'avez jamais plus de 2-3 issues à traiter à la fois.

Démarche de travail recommandée

Étape 0 Setup & scaffold

  • Créer le dépôt Git (un par binôme), branche main.
  • Initialiser le projet vide (spring init, npm init…), commit du scaffold.
  • Démarrer SonarQube local, créer le projet f1-binome-XX, générer un token.
  • Configurer l'outil de coverage adapté au langage.
  • Définir le "New Code" sur Number of days = 1 AVANT le premier scan (sinon le gate sera trivialement vert).

Étape 1 Premier scan + design rapide

  • Lancer un scan Sonar sur le scaffold vide → vous devez obtenir le badge Passed (0 LOC, 0 issue).
  • Décider à deux du découpage : packages/dossiers, séparation controller/service/repository, modèle de données.
  • Esquisser sur papier ou sur un fichier DESIGN.md les signatures des principales méthodes métier (calcul du prix, application des remises, vérification de disponibilité, calcul du remboursement).
  • Consulter le contrat d'API de référence (openapi.yaml ou swagger.html dans le dossier de l'évaluation) pour aligner vos formats de requête/réponse.

Comment se répartir le travail en binôme

Plusieurs stratégies possibles. Choisissez celle qui colle à vos forces respectives, mais évitez de vous découper "moitié features chacun" sans coordination : vous allez créer des conflits Git et casser la cohérence du moteur de prix.

Option 1 — Par couche
  • Dev 1 : services métier (phase C moteur de prix, phase E remboursement) + tests unitaires associés
  • Dev 2 : controllers / endpoints + persistance + tests d'intégration

Idéal si l'un est plus fort en logique métier, l'autre en API/infra.

Option 2 — Par domaine
  • Dev 1 : catalogue (tribunes + sessions + spectateurs — phase B)
  • Dev 2 : transactions (moteur de prix + réservation + annulation — phases C + D + E)

Chacun a sa "verticale" propre. Bon pour minimiser les conflits Git.

Option 3 — Pair / Switch
  • Phases A et B faites en pair-programming (un clavier, l'autre revue)
  • À partir de la phase C, l'un code la feature, l'autre écrit ses tests, puis on inverse

Plus lent en LOC/h mais qualité supérieure et apprentissage croisé.

Quelle que soit l'option choisie : nommez un référent qualité (celui qui surveille le dashboard Sonar et alerte le binôme dès qu'une issue apparaît), et un référent Git (celui qui orchestre les merges, résout les conflits, garde un historique propre). Ces deux rôles peuvent être portés par la même personne, ou tourner à chaque phase.

Étape 2 Implémentation itérative (cœur du devoir)

Appliquer la boucle de la section "Méthode". L'ordre conseillé ci-dessous suit une logique de dépendances : on valide les fondations avant la logique métier, on isole le moteur de prix (le morceau le plus testable) avant de le brancher au reste, et on ne touche aux statistiques qu'une fois qu'il y a des données à agréger.

A Fondations

Ce que vous faites : classes/structs des 4 entités (Grandstand, Session, Spectator, Reservation) ; couche de persistance en mémoire (Map / Dict par entité, compteurs d'ID) ; helpers réutilisables (validation email, calcul d'âge à partir d'une date de naissance, comparaison de dates).

À tester : les helpers — surtout le calcul d'âge sur des cas limites (anniversaire à venir / passé, leap year) et la validation email.

Pourquoi en premier : socle de tout le reste, et le code des helpers est pur (zéro dépendance) donc facile à tester proprement dès le départ.

B CRUD plat (features 1 à 4)

Ce que vous faites :

  • POST + GET tribunes (feature 1, 2) avec filtres category et covered
  • POST + GET sessions (feature 3) avec multiplicateur par défaut selon le type
  • POST + GET spectateurs (feature 4) avec unicité email et validation date de naissance

À tester : chaque endpoint en happy path + au moins 2 cas d'erreur de validation (champ manquant, format invalide, doublon email).

Pourquoi maintenant : sans données en base, rien à tester en aval. Cette phase consolide aussi votre architecture en couches (controller → service → repository).

C Moteur de prix (feature 5) — le cœur testable

Ce que vous faites : un service dédié qui prend en entrée (tribune, sessions, nb places, spectateur) et retourne un objet "détail du prix" (sous-total, remises appliquées, total). Incrémenter une règle à la fois :

  • C.1 — Prix de base (règles 1 + 2) : prix_tribune × multiplicateur_session × nb_places, sommé sur toutes les sessions
  • C.2 — Pass weekend (règle 3) : −20% si FRIDAY + SATURDAY + SUNDAY
  • C.3 — Fidélité (règle 4) : −5% SILVER / −10% GOLD, cumulable après le pass weekend
  • C.4 — Tarif jeune (règle 5) : −50% si < 16 ans
  • C.5 — Endpoint POST /reservations/quote : exposer le service via l'API

À tester : un test unitaire par règle (nommé explicitement, ex : applyWeekendPassDiscount_when3DaysSameGrandstand) + un ou deux tests combinatoires (pass weekend + GOLD, jeune + fidélité).

Pourquoi maintenant : ce service est une fonction pure — zéro dépendance à la persistance — donc testable à 100% en isolation. C'est aussi le morceau le plus susceptible de générer de la complexité cyclomatique : le faire en premier et le garder propre évite qu'il devienne une god function.

D Réservation (feature 6)

Ce que vous faites : POST /reservations qui enchaîne les vérifications dans cet ordre :

  1. Validation des entrées (existence du spectateur, de la tribune, des sessions)
  2. Max 6 places (règle 6)
  3. Disponibilité sur chaque session (règle 7) — helper "places restantes sur tribune × session"
  4. Appel au moteur de prix de la phase C
  5. Persistance + retour de la réservation créée

À tester : un test de refus par règle d'éligibilité (400/409 attendus) + un happy path qui crée une réservation et vérifie que le prix correspond au quote.

Pourquoi maintenant : nécessite le moteur de prix (phase C) et le CRUD entités (phase B). Aucune raison de coder ça avant.

E Annulation (feature 7)

Ce que vous faites :

  • E.1 — Service isolé "calcul du remboursement" (règle 8) qui prend la date de la première session et la date du jour, retourne le taux (100% / 0%) et le montant
  • E.2 — POST /reservations/{id}/cancel : passer le statut à CANCELLED, enregistrer le montant remboursé, libérer les places sur les sessions concernées

À tester : les 2 paliers temporels (J−15 et J−3), le refus de double-annulation, la libération effective des places (rejouer une réservation après annulation doit marcher).

Pourquoi maintenant : besoin de réservations CONFIRMED existantes pour pouvoir les annuler. Service de remboursement isolé pour les mêmes raisons que le moteur de prix : fonction pure, facile à tester.

Rescannez Sonar régulièrement (typiquement après chaque phase A → E, ou plus souvent si vous en sentez le besoin). Si le Quality Gate tombe en Failed, arrêtez d'avancer et corrigez avant de passer à la phase suivante.

Étape 3 Finalisation : couverture, polissage, vérif fonctionnelle

  • Compléter les tests manquants pour atteindre ≥ 4% de couverture sur le code produit.
  • Traiter les Security Hotspots restants (les marquer "reviewed" si justifié, ou les corriger).
  • Dernier scan : viser Quality Gate Passed définitif.
  • Recette manuelle : exécuter un scénario de bout en bout (créer une tribune GOLD + 3 sessions du weekend, inscrire un spectateur SILVER, simuler le prix d'un pass weekend, créer la réservation, l'annuler à J−15 et vérifier le remboursement à 50%) et noter le résultat.

Étape 4 Rapport final

  • Finaliser le rapport (cf. section Livrables).
  • Capturer les écrans Sonar finaux (Overview, Measures, Quality Gate Passed).

Livrables

À déposer dans une archive ZIP nommée MSPR_QG_NOM1-NOM2.zip contenant :

  1. Code source complet du projet (sources, tests, fichier de build, README de démarrage).
  2. Historique Git — sortie de git log --oneline --decorate dans un fichier git-log.txt. L'historique doit être suffisamment cohérent pour pouvoir rejouer plusieurs scans Sonar à différents moments du projet et constater l'évolution du Quality Gate.
  3. Rapport (format libre) structuré ainsi :
    • Page de garde : noms du binôme, date, stack choisie, lien dépôt.
    • §1 Architecture du projet — schéma simple (couches + entités) et justification du découpage.
    • §2 Implémentation des règles métier — un mini-tableau "règle → où dans le code → tests qui la couvrent".
    • §3 Métriques Sonar finales — captures du dashboard + tableau (LOC, Coverage, Duplications, Complexity, Quality Gate).
    • §4 Historique qualité — au moins 3 captures du Quality Gate prises à différents moments du projet (début, milieu, fin) montrant l'évolution.
    • §5 Recette manuelle — scénario de bout en bout joué + résultats.
    • §6 Bilan critique — difficultés rencontrées, règles Sonar contestables, ce que vous referiez autrement.
  4. Captures Sonar du dashboard final, au minimum : Overview avec badge Passed, Measures (Coverage + Duplications + Complexity), Issues (idéalement vide).

Pour aller plus loin — bonus optionnels, non notés

Ces pistes ne rapportent pas de points au barème, mais sont bienvenues si vous avez le temps — et particulièrement pertinentes à mentionner dans le §6 du rapport (bilan critique). Elles prolongent naturellement la démarche "Clean as You Code" en automatisant la discipline.

Conventional Commits + commitlint

Imposez automatiquement la convention type(scope): message sur tous vos commits — un hook bloque les messages mal formés avant qu'ils n'entrent dans l'historique.

Stack : @commitlint/cli + @commitlint/config-conventional + husky (Node — fonctionne aussi pour piloter des projets non-Node).

Pre-commit hook qualité

Lancez automatiquement le linter (ou le formateur de code) et les tests avant chaque commit. Si quelque chose échoue, le commit est bloqué — vous ne pouvez plus pousser de la dette technique par inadvertance.

Stack : husky (JS/TS), pre-commit (Python), lefthook (multi-langage).
